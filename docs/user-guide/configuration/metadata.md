# Metadata Field Documentation

The `metadata` section provides essential information about your PrompTrek configuration file. This metadata helps organize, version, and document your prompt files.

## Overview

Metadata is **required** in all PrompTrek schema versions and serves several purposes:

- **Documentation**: Describe the purpose and scope of your prompt file
- **Version Control**: Track versions and changes over time
- **Organization**: Categorize and tag prompt files
- **Attribution**: Credit authors and maintainers
- **Discovery**: Help others find and understand your configurations

## Required Fields

### title

**Type**: `string` (required)

A concise, human-readable name for your prompt file.

```yaml
metadata:
  title: "Full-Stack TypeScript Project"
```

**Best Practices**:

- Keep it short and descriptive (2-5 words)
- Use title case
- Describe the project or domain
- Be specific enough to differentiate from other configs

**Examples**:

```yaml
# Good
title: "Python FastAPI Backend"
title: "React TypeScript Frontend"
title: "Data Science Toolkit"

# Avoid
title: "My Project"              # Too vague
title: "config"                  # Not descriptive
title: "The Amazing TypeScript Full-Stack Web Application With React and Node.js"  # Too long
```

### description

**Type**: `string` (required)

A brief description of what this prompt file configures or its purpose.

```yaml
metadata:
  description: "AI assistance for a TypeScript full-stack application with React frontend and Node.js backend"
```

**Best Practices**:

- One to two sentences maximum
- Explain the purpose or scope
- Mention key technologies or frameworks
- Describe what makes this configuration unique

**Examples**:

```yaml
# Good examples
description: "Python FastAPI backend with PostgreSQL, focusing on clean architecture and comprehensive testing"
description: "React TypeScript frontend using Next.js, emphasizing performance and accessibility"
description: "Data science project with Jupyter notebooks, focusing on reproducible research and visualization"

# Helpful context
description: "Monorepo configuration for microservices architecture with shared TypeScript libraries"
description: "Legacy Java Spring Boot application undergoing modernization to Spring Boot 3"
```

## Optional Fields

### version

**Type**: `string` (optional)

Semantic version of your prompt configuration (not the schema version).

```yaml
metadata:
  version: "2.1.0"
```

**Best Practices**:

- Follow [semantic versioning](https://semver.org/): `MAJOR.MINOR.PATCH`
- Increment MAJOR for breaking changes
- Increment MINOR for new features
- Increment PATCH for bug fixes or minor updates
- Keep in sync with your project version if applicable

**Examples**:

```yaml
# Project version tracking
version: "1.0.0"    # Initial release
version: "1.1.0"    # Added new commands
version: "1.1.1"    # Fixed typo in guidelines
version: "2.0.0"    # Restructured entire configuration

# Development stages
version: "0.1.0"    # Alpha/early development
version: "0.9.0"    # Beta/pre-release
version: "1.0.0"    # Production ready
```

### author

**Type**: `string` (optional)

Name or email of the configuration author or maintainer.

```yaml
metadata:
  author: "Development Team <dev@example.com>"
```

**Formats**:

```yaml
# Name only
author: "Jane Doe"

# Email only
author: "dev@example.com"

# Name and email
author: "Jane Doe <jane@example.com>"

# Team
author: "Platform Engineering Team"

# Organization
author: "Acme Corp Engineering"
```

### created

**Type**: `string` (optional)

ISO 8601 date when the configuration was created (`YYYY-MM-DD`).

```yaml
metadata:
  created: "2024-01-15"
```

**Best Practices**:

- Always use ISO 8601 format: `YYYY-MM-DD`
- Don't update this field after creation
- Can be auto-generated by `promptrek init`

```yaml
# Correct
created: "2024-01-15"
created: "2023-12-01"

# Incorrect
created: "1/15/2024"           # Wrong format
created: "15-01-2024"          # Wrong format
created: "January 15, 2024"    # Wrong format
```

### updated

**Type**: `string` (optional)

ISO 8601 date when the configuration was last updated (`YYYY-MM-DD`).

```yaml
metadata:
  updated: "2024-03-20"
```

**Best Practices**:

- Update whenever significant changes are made
- Use ISO 8601 format: `YYYY-MM-DD`
- Can be automated in CI/CD workflows
- Should be ≥ `created` date

```yaml
metadata:
  created: "2024-01-15"
  updated: "2024-03-20"        # Later than created
```

**Automation Example**:

```bash
# Update 'updated' field automatically
sed -i "s/updated:.*/updated: \"$(date +%Y-%m-%d)\"/" project.promptrek.yaml
```

### tags

**Type**: `array of strings` (optional)

Optional tags for categorization, search, and organization.

```yaml
metadata:
  tags: [typescript, react, node, testing, ci-cd]
```

**Best Practices**:

- Use lowercase
- Separate words with hyphens (kebab-case)
- 3-7 tags is optimal
- Include:
    - Programming languages
    - Frameworks
    - Domains or categories
    - Tools and technologies

**Examples**:

```yaml
# Backend project
tags: [python, fastapi, postgresql, rest-api, backend]

# Frontend project
tags: [typescript, react, nextjs, frontend, ssr]

# Full-stack
tags: [typescript, react, node, fullstack, mongodb]

# Data science
tags: [python, jupyter, pandas, machine-learning, data-viz]

# DevOps
tags: [kubernetes, docker, terraform, ci-cd, devops]

# Mobile
tags: [react-native, typescript, mobile, ios, android]
```

**Tag Organization**:

```yaml
tags:
  # Languages
  - typescript
  - python

  # Frameworks
  - react
  - fastapi

  # Categories
  - fullstack
  - microservices

  # Features
  - testing
  - ci-cd
```

## Complete Examples

### Minimal Metadata

The absolute minimum required:

```yaml
metadata:
  title: "My Project"
  description: "Project description"
```

### Standard Metadata

Typical metadata for a production project:

```yaml
metadata:
  title: "E-Commerce Platform"
  description: "Microservices-based e-commerce platform with React frontend and Node.js services"
  version: "2.3.0"
  author: "Platform Team <platform@example.com>"
  created: "2023-06-01"
  updated: "2024-01-15"
  tags: [typescript, react, node, microservices, e-commerce]
```

### Comprehensive Metadata

Full metadata with all optional fields:

```yaml
metadata:
  title: "Healthcare Data Platform"
  description: "HIPAA-compliant healthcare data platform with real-time analytics and patient management"
  version: "3.1.2"
  author: "Healthcare Engineering Team <health-eng@example.com>"
  created: "2022-03-15"
  updated: "2024-01-20"
  tags:
    - typescript
    - react
    - node
    - healthcare
    - hipaa
    - analytics
    - real-time
```

## Use Cases

### Project Evolution Tracking

Track your project's evolution through metadata:

```yaml
# Initial version
metadata:
  title: "Simple Todo App"
  description: "Basic todo application"
  version: "0.1.0"
  created: "2023-01-01"
  tags: [javascript, react]

# After growth
metadata:
  title: "Task Management Platform"
  description: "Enterprise task management with real-time collaboration and analytics"
  version: "2.5.0"
  created: "2023-01-01"
  updated: "2024-01-15"
  tags: [typescript, react, node, websockets, enterprise, collaboration]
```

### Multi-Team Projects

Indicate team ownership:

```yaml
# Backend team
metadata:
  title: "API Services"
  description: "REST and GraphQL API services"
  author: "Backend Team <backend@company.com>"
  tags: [node, graphql, rest-api, backend]

# Frontend team
metadata:
  title: "Web Application"
  description: "React-based web application"
  author: "Frontend Team <frontend@company.com>"
  tags: [react, typescript, frontend]

# Infrastructure team
metadata:
  title: "Infrastructure as Code"
  description: "Terraform and Kubernetes configurations"
  author: "DevOps Team <devops@company.com>"
  tags: [terraform, kubernetes, infrastructure]
```

### Domain-Specific Projects

metadata:

## Validation

PrompTrek validates metadata:

```bash
promptrek validate project.promptrek.yaml
```

**Common validation errors**:

```
❌ metadata.title is required
❌ metadata.description is required
❌ metadata.created: Invalid date format (use YYYY-MM-DD)
❌ metadata.updated: Must be >= created date
```

## Best Practices

!!! tip "Be Descriptive"
    Use `description` to provide context that helps both humans and AI understand your configuration's purpose.

!!! tip "Keep Versions Updated"
    Increment `version` when making significant changes to help track evolution and compatibility.

!!! tip "Tag Thoughtfully"
    Choose tags that make your configuration discoverable and categorizable. Include languages, frameworks, and domains.

!!! tip "Track Dates"
    Use `created` and `updated` to understand configuration age and maintenance status.

!!! warning "Don't Over-Tag"
    Too many tags (>10) dilute their usefulness. Focus on the most important categories.

!!! note "Automation Friendly"
    Structure metadata to support automation:
    ```bash
    # Extract version
    grep "version:" project.promptrek.yaml | cut -d'"' -f2

    # Update date automatically
    sed -i "s/updated:.*/updated: \"$(date +%Y-%m-%d)\"/" project.promptrek.yaml
    ```

## Schema Reference

Metadata structure in JSON Schema:

```json
{
  "PromptMetadata": {
    "type": "object",
    "required": ["title", "description"],
    "properties": {
      "title": { "type": "string" },
      "description": { "type": "string" },
      "version": { "type": "string" },
      "author": { "type": "string" },
      "created": { "type": "string" },
      "updated": { "type": "string" },
      "tags": {
        "type": "array",
        "items": { "type": "string" }
      }
    }
  }
}
```
